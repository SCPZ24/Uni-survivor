# mainline-guide：怎么搭例题主线

本文件是 `01-mainline.md` 的写作细则。骨架见 `../templates/mainline.md`。

## 一、选题：最小但完备

目标是一个**集合属性**：用**尽量少**的例题，让它们的**并集**覆盖 SKILL.md 第 1 步抽出的全部应考知识点。

操作要点：

- **先铺满，再瘦身。** 先确认候选例题组能盖住清单上每一项；再砍掉「覆盖完全被其他题包含」的冗余题。
- **能串则串。** 如果一道稍复杂的综合题能自然路过 3 个点，就用它替掉 3 道各讲 1 个点的小题——前提是讲解仍然清楚、不牵强。
- **不牵强。** 串点是为了少而全，不是硬塞。若硬串会让解题逻辑别扭，就拆成 1:1 的多道小题。
- **可以「编」题。** 课件没有现成例题时，自行编一道贴合考点、难度贴近考试的题。编题优先于「为了凑题而漏点」。
- **难度对齐考试。** 面向考前突击，例题难度贴近真实考题，不做纯科研难度的拓展。

## 二、1:1 vs 1:多 的判断与处理

对**每一道题**单独判断它覆盖一个还是多个知识点——**不要**先给整门课定性。

- **1:1（常见于数值分析、算法、离散数学的单点计算题）：** 一题一个点。解答聚焦把这一个点讲透：原理 + 标准步骤 + 易错。
- **1:多（常见于操作系统、计算机网络、数据库的综合/流程题）：** 一题串多个点。解答里每路过一个点就**显式停一下**，用小标题或提示块点出「这里用到的是 XX 知识点」，再讲透，然后继续往下解。
- 覆盖表的「对应知识点」列要如实反映：1:1 就写一个，1:多就把这道题路过的点**全部列出**。

## 三、引导式解答的写法

每道例题不是「给答案」，而是**带着读者走一遍思考**，让知识点在解题过程中被讲透。每题建议含这些层次（按需取舍，见 `templates/mainline.md`）：

1. **题目**：完整题面。
2. **考点定位**：一句话点明这题路过哪些知识点（与覆盖表呼应）。
3. **思路引导**：先问「看到这种题先想什么 / 怎么判断该用哪个方法」，再展开。讲**为什么这么做**，不只给步骤。
4. **就地讲透**：在用到某概念 / 公式时，当场把它的定义、适用条件、推导要点讲清楚——读者读这一题就不必再翻别处。
5. **完整解答**：清晰的步骤与结果。
6. **易错点 / 坑**：这类题学生最常错在哪、判分点在哪、边界情况怎么处理。

写作语气：面向考前突击的学生，直给、可操作，省去客套。

## 四、视觉强调规则

用视觉层级让「重点」一眼可见，由你自行判断轻重：

- **加粗**：关键结论、必背公式名、判分点、易错警告的核心词。
- `行内代码`：变量、符号、短公式片段。
- 代码块 / 公式块：成段的推导、算法步骤、较长公式。
- 提示块（引用 `>`）：用约定前缀分类——
  - `> ⚠️ 易错：` 常见错误 / 扣分点
  - `> 💡 技巧：` 应试技巧、速算、记忆法
  - `> 📌 考点：` 高频考点提醒
- 不要通篇加粗——满屏重点等于没有重点。一道题里高亮控制在真正关键的几处。

## 五、头部覆盖表怎么填

覆盖表是 mainline 的**脊柱**，放在文件最前面。它同时承担三件事：① 覆盖脊柱（自上而下读一遍就能看出这组题是否铺满考纲）② 导航 ③ 显式呈现 1:1 / 1:多 关系。

至少两列：

| 列 | 内容 |
|----|------|
| 题号 | 例题编号，做成指向该题锚点的链接，承担导航 |
| 对应知识点 | 这道题路过的知识点；1:1 写一个，1:多 把多个**全列出** |

**关键：必须让「整张考纲的覆盖去向」一目了然。** 第 1 步抽出的清单里，有些知识点没被任何例题覆盖、只进了 cheatsheet——这些**绝不能消失**。处理方式：在覆盖表正下方紧跟一个「**仅见于 cheatsheet 的知识点**」小节，把这些点逐条列出并注明「→ 见 02-cheatsheet.md」。

这样读者扫一眼头部就能回答：「考纲上每个点到底进了 mainline 还是 cheatsheet？」——不留任何「看不出某考点有没有被收录」的盲区。

填表自检：把 SKILL.md 第 1 步清单的每一项，要么对到表里某个题号，要么对到「仅见于 cheatsheet」小节。两边都没有 = 漏了，回去补。
